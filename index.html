What are accessible names?

Accessible names are used by assistive technologies to identify elements in the accessibility tree.

They help assistive technology users to understand the purpose of relevant UI components.

You can see the name assigned to each accessible object by viewing the accessibility tree.

Do these two concepts have to match?

In some cases, the accessibility name matches the visible text label.

Example 02

However, it is possible to create an accessible name that is partially or even entirely different from the visible text label.

For example, the accessible name could include additional content before the visible text label.

Example 03

Or after the visible text label.

Example 04

It is possible to present entirely different content in the accessibility tree.

Example 05

However, this is not a good practice. Accessible names should always match some aspect of the visible text label.

A mismatch between the visible text label and the accessible name could confuse some users.

For example, some sighted users also use screen readers. If the visible text label doesn’t match the accessible name, this could be confusing.

This mismatch could also be confusing if assistive technology users need to describe aspects of the UI to sighted users and vice versa.

  • Accessible Name and Description Computation 1.1
  • HTML Accessibility API Mappings 1.0

Let’s look at how names and descriptions are calculated for some example elements.

<a> and <button> elements

In the case of <a> and <button> elements, the accessible name is defined in the following order:

  1. If present, use aria-labelledby.
  2. Otherwise, if present, use aria-label.
  3. Otherwise, if present, use elements content.
  4. Otherwise, if present, use title.
  5. Otherwise, there is no accessible name.

We’re now going to look at a <button> example with all of these accessible name options applied.

  • The aria-labelledby has a value: “One”
  • The aria-label has a value: “Two”
  • The <button> content is: “Three”
  • The title has a value: “Four”

We’ll then remove each accessible name option one-by-one until we’re left with no accessible name.

Unlike the title, it can be presented on-screen and is therefore available to all users.

The shifting title

As we move through the examples, you’ll notice that the title attribute is theoretically the last resort for an accessible name.

The title attribute is also used as a last resort for an accessible description, but only if two conditions are met:

  1. The title has not already been used for the accessible name.
  2. There is no aria-describedby present.

If other accessible name options are present along with the title, this value could become the accessible description.

Example 09

However, the title attribute should only ever be used as an accessible name for iframe elements.

And, the title attribute should never be used as an accessible description for elements as it is highly inaccessible.

<input> and <textarea> elements

In the case of <input> and <textarea> elements, the accessible name is defined in the following order:

  1. If present, use aria-labelledby.
  2. Otherwise, if present, use aria-label.
  3. Otherwise, if present, use <label> content.
  4. Otherwise, if present, use title.
  5. Otherwise, if present, use placeholder.
  6. Otherwise, there is no accessible name.

The <select> element uses a similar order but does not include the placeholder step.

Problems with the placeholder

Despite the clearly defined rules for calculating names, the placeholder beats the title in most browser and screen reader combinations.

Example 20

Also, if an accessible name is in place, the placeholder is often still announced in most browser and screen reader combinations.

In some browser and screen reader combinations, the placeholder is announced as part of the accessible name.

In other browser and screen reader combinations, it is announced as part of the accessible description.

This is just one of many reasons why the placeholder should never be used as the only visible text label or accessible name for elements.

<img> elements

In the case of <img> elements, the accessible name should be defined in the following order:

  1. If present, use aria-labelledby.
  2. Otherwise, if present, use aria-label.
  3. Otherwise, if present, use alt.
  4. Otherwise, if present, use title.
  5. Otherwise, there is no accessible name.

Whenever possible, always try to use the elements alt attribute as the accessible name, as it is available to all users.

And the accessible description should be defined in the following order:

  1. If present, use aria-describedby.
  2. Otherwise, if present, use title.
  3. Otherwise, there is no accessible description.

If there is an alt attribute in place for the accessible name, an accessible description should not be required.

<fieldset> elements

In the case of <fieldset> elements, the accessible name should be defined in the following order:

  1. If present, use aria-labelledby.
  2. Otherwise, if present, use aria-label.
  3. Otherwise, if present, use <legend>.
  4. Otherwise, if present, use title.
  5. Otherwise, there is no accessible name.

Whenever possible, always try to use the <legend> elements content as the accessible name, as it is available to all users.

And the accessible description should be defined in the following order:

  1. If present, use aria-describedby.
  2. Otherwise, if present, use title.
  3. Otherwise, there is no accessible description.

Issues with the <fieldset>

In theory, a description could be applied to the overall <fieldset> via an aria-describedby attribute.

In the following example, a radio group has an error message associated with the <fieldset> via an aria-describedby.

<fieldset aria-describedby="e01">
  <legend>Do you like boats?</legend>
  <div>
    <input type="radio" id="y" name="b">
    <label for="y">Yes</label>
  </div>
  <div>
    <input type="radio" id="n" name="b">
    <label for="n">No</label>
  </div>
  <p id="e01">Error: Choose at least one option</p>
</fieldset>

However, this method is not consistently supported across all browser and screen reader combinations.

Currently, the most robust method of associating an error message with the radio group is to apply the aria-describedby directly to each <input> element.

<fieldset>
  <legend>Do you like boats?</legend>
  <div>
    <input type="radio" id="y" name="b"
    aria-describedby="e01">
    <label for="y">Yes</label>
  </div>
  <div>
    <input type="radio" id="n" name="b"
    aria-describedby="e01">
    <label for="n">No</label>
  </div>
  <p id="e01">Error: Choose at least one option</p>
</fieldset>
